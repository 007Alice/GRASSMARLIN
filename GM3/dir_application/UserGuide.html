1 GrassMarlin Overview

GrassMarlin is an open-source software tool that generates a visualization of a network to facilitate further analysis. There are many tools that fill this role in different ways, so it is important to understand the role that GrassMarlin is intended to fill and this is easily framed by identifying what it does not do. Often GrassMarlin is mistakenly believed to perform analysis, but it is a tool to organize data for analysis, not to perform the analysis directly. GrassMarlin is also not a data acquisition tool; certain versions can gather data from a NIC running in promiscuous mode, but otherwise has no capture capabilities. GrassMarlin is not an IDS, nor is it a replacement for WireShark.

GrassMarlin maps the connections between endpoints on a network to illustrate the flow of data. The visualization allows the user to quickly sift through a large volume of detailed data to find items of interest. Frequently, GrassMarlin is used to either validate a network diagram or to establish a network diagram when none is readily available.

Originally, GrassMarlin was developed with a focus on Supervisory Control and Data Acquisition (SCADA) and Industrial Control System (ICS) hosts. It has, however, expanded from these origins to operate in a broader set of environments. With the plugin support added in version 3.3.0, GrassMarlin is adaptable to nearly any network environment, to include wireless and serial. With the right set of plugins, it could probably support analysis of social networks as well.

You'll find notes like this one throughout this document; these are asides from the developers intended to add context and/or entertainment to what is otherwise a very dry document. If you think reading this is bad, you should try writing it.

GrassMarlin accepts as input PCAP files as well as router and switch configurations to produce a Physical Graph and Logical Graph.

The Physical Graph is built by identifying physical endpoints (generally MAC addresses) with the goal of diagramming the physical connections between devices. On a switched network, it is generally impossible to determine how devices within a switched segment connect to each other--this is where switch configuration files come into play. These configuration files allow the otherwise ambiguous topology to be mapped. New to GrassMarlin 3.3.0, the contents of the Physical Graph can be inferred from PCAP data. Switching architecture remains an unknown, but network segments can be identified even in the absence of switch configuration files.

The Logical Graph is built by identifying logical endpoints (generally IP + MAC pairs) to produce a diagram of logical communication channels.

In prior versions of GrassMarlin, imported packet data would be subjected to evaluation by a set of Fingerprints to identify known protocols, extracting relevant details to display to the user. This is still possible in GrassMarlin 3.3.0 (and, indeed, is the default) however the data processing pipeline is now exposed to the user and may be modified to change how data is processed. New Plugins can change how data flows through the application, adding, removing, or modifying information as deemed appropriate by the analyst--this is, after all, a tool meant to facilitate work by analysts, not replace them.

This is the last planned release of GrassMarlin. The greatly expanded plugin model, coupled with the abstractions in the core data model, allow us to shift focus from writing the core application framework to developing solutions for specific problems. These solutions are expected to take the form of plugins. Every once in a while a plugin requires a change to the core behaviors, but as time goes on that happens less and less. While the GrassMarlin platform may be at the end of its development path, Plugins to meet specific needs are expected to take over the development schedule.

1.1 Versioning

Releases of GrassMarlin are labeled with a 3-part version number and a revision. This is formally written as Major.Minor.PatchrRevision. Most often, the shortened form of Major.Minor is used.

The Major version number indicates the set of features which are considered to be desirable--that is, the features that are intended to be present. Version 3, for example, is striving to better accomodate data sources while enabling greater analytic potential. By contrast, earlier versions were aimed at establishing a basic proof-of-concept for core features.

The Minor version number indicates the set of features which are actually present--features may be absent because of errors in programming, a need to gather more data before implementation, or because the developers simply need more time to produce a feature.

The fill feature set of a Major version isn't actually known when we start work on that version. We know the general goal of the 3.X codebase is to accept more forms of data and allow more interaction and organization with that data, but we can't conclusively say yet exactly how that will take form. Feedback from the 3.0 release had a significant impact on the features that went into 3.2 (3.1 was never publicly released and was only a small update to 3.0). As 3.3 very thoroughly addresses the goals that were set for the 3.X version, there is unlikely to be a 3.4--there will be patches to 3.3 and plugins, or a 4.0 if a new direction is chosen.

If there is a bug in a particular Minor version, then a Patch is issued to address this.

In the past, patches were seldom released due to the overhead associated with a public release. We have done a lot to reduce the overhead and are better positioned to issue patches than we were in the past. The 3.1 release, for example, languished in red tape for nearly a year before being abandoned. The 3.3.0Beta2 release took 1 week from when a stable build was established.

The Revision number corresponds to the exact set of files in the master source repository that was used to build the released version.

As part of the process improvements, 3.3 was built on a different network than its predecessors. As a result, it is decoupled from the source control previously used and the revision number was reset.

1.2 Supported Platforms

Supported platforms are those on which every effort has been made to ensure the GrassMarlin application installs and executes as intended. Failure for GrassMarlin to start on these platforms with an out-of-the-box configuration is regarded as a bug.

  • Microsoft Windows (64-bit 7, 8, and 10)
  • Fedora (23)
  • Ubuntu (14.04, 15.10, and Security Onion)
  • Kali 2.0
  • CentOS (6, 7)
  • Debian (8)

Additionally, installers are provided for some unsupported platforms (32-bit Windows, for example). In these cases we have tested the application in these environments, however there are no assurances it will work without additional configuration. We do not formally provide support for these platforms, but we have observed working installations. Running GrassMarlin on these platforms may result in degraded performance and/or depend on post-install reconfiguration, but is possible.

The GrassMarlin code does work on Windows XP, and the Windows installers should also run on Windows XP, however the typical Windows XP Java installation is not fully compatible with GrassMarlin, requiring some minor changes to the GrassMarlin batch file to compensate. the necessary changes vary by specific circumstances, but can generally be inferred from the error messages and an Internet search of Java errors. There is no support for this, but it has been made to work in specific environments.

1.3 Support

There are several distinct tiers of support that are made available. While this application is distributed to the global public on GitHub, it is a product of the United States Government and is also available to government agencies through other channels.

General Q&A and support regarding execution on a supported platform is available through GitHub for the general public. We can only support English-language queries.

Government partners may have access to additional channels for support, depending on what authorities and agreements apply.

Agency partners should contact the GrassMarlin team lead directly.

The GrassMarlin team is small--small enough that we all get sick on doughnut day when management brings us a dozen doughnuts. We have nowhere near the resources we would like to have to support GrassMarlin (except doughnuts--we have plenty of those), to say nothing of the resources users would like us to have. We do our best to ensure a quality release and provide support, but the simple reality is that we don't have the manpower for a well-polished product. We're a tiny prototype team and we aim for a 90% solution that gets the job done well enough.

2 GrassMarlin Structure

In order to describe the structure of the user interface, it helps to understand the terminology associated with the application.

GrassMarlin 3.3 is a significant conceptual departure from earlier versions, although the presentation is intended to be similar. The processing pipeline is exposed so that the user may change how it behaves and support for Plugins allows for significant modification to existing behaviors as well as the addition of others. For this reason what is described here is the core of GrassMarlin 3.3 only--as it runs with no plugins. Where relevant, behaviors that may be created with official plugins are noted.

The main window of GrassMarlin, depicted below, is divided into three major areas, all of which may be resized by dragging the border between sections. The right side of the screen contains the Visualizations while the left is split between Navigation on the top and the Message Log on the bottom. Each of these is described in detail later.

Figure 1: New GrassMarlin Session

Beta versions display a "β" in the center of the Visualization to reinforce the fact that it is a beta build and things may not work. Additionally, it is a reminder to the developers that we're working in a Beta branch. This particular screenshot also includes a tab titled "Visual Pipeline" which is created by the iadgov.visualpipeline Plugin--this is a Plugin meant to be used by developers to assist in troubleshooting new Pipeline Stages.

The initial display is blank, as no data has yet been Imported. When data is Imported, it is added to the current Session. Most often, the data that is Imported comes from files, and these files are interpreted by Parsers which feed information from the files into the Pipeline.

The Pipeline contains one or more Entry Points and Stages. Each Entry Point points to a Stage, and when a file is Imported the interpreted data is sent to the Stage associated with a chosen Entry Point. The data flowing through the Pipeline is called the Data Stream. The Stages perform many different actions, to include removing certain elements from the Data Stream, forking the data to different paths through the Pipeline, adding new data objects to the Data Stream, or modifying the Session.

GrassMarlin includes a single default pipeline which has two entry points; a default which retains packet metadata and one that discards it.

Figure 2: Pipeline Editor depicting the Default Pipeline

To modify the GrassMarlin user experience, Plugins may be loaded. Plugins are loaded when the application launches and can produce many different effects, most frequently support for new input types and the addition of new Pipeline Stages. It is possible to suppress loading Plugin files (technically, Plugin loading is suppressed unless GrassMarlin is explicitly instructed to load them), however there is an internal Plugin titled "Grassmarlin" that is always loaded.

It is a little odd to treat what had been core behaviors as optional blocks of functionality, however it eliminated a lot of the special-casing that 3.2 required and acted as a check to ensure clear lines of separation between functional elements. The internal Plugin is actually a collection of other Plugins, all of which act on their own, but are loaded under the "Grassmarlin" label.

2.1 Message Log

The message log displays a list of messages generated by the application describing the actions it has been instructed to perform, the outcome of those actions, errors encountered, and other potentially-useful bits of information. All messages are also written to a log file, one file per launch of the GrassMarlin application. Each Message is associated with a Severity. Generally, Messages which are marked as Information, Completion, Warning, and Error are sent to the Message Log while others are suppressed. All messages are sent to the log file.

The only other Severity that is presently defined is "Pedantic Developer Spam" which is so-named because it is generally used to provide excessively verbose details to assist in troubleshooting. These messages are of minimal use to a user, but are useful when attempting to diagnose odd behaviors. These messages will be displayed in the Message Log when the GrassMarlin application is running in Developer Mode.

2.2 Visualization

The Visualization area contains multiple tabs which each provide some form of visualization of the data contained in the Session. The internal Plugin provides a Logical Graph and Physical Graph, with an option to display the Reports associated with the Logical Graph. Official Plugins also provide the Visual Pipeline tab, which displays the current Pipeline with detailed information describing the current state of the Pipeline. It can be informative to use, but it is intended to be used to diagnose errors in Pipeline and Stage design and implementation.

There used to be a "Sniffles" tab. The content displayed there is now integrated into the Logical and Physical Views, and so it has been removed. In theory. In practice, we haven't written the Plugin that provides the necessary Import functionality to make this happen.

The Logical Graph allows for the creation of Watch and Filtered Views, which are each allocated their own tab. The Logical Graph, to include Watch and Filtered Views as well as the Physical Graph are described in detail in their own sections.

2.3 Navigation

The contents of the Navigation area change as the active tab of the Visualization area is changed. Each Tab of the Visualization specifies its own Navigation control to be used. For this reason details are relegated to the sections describing each Visualization.

2.4 Pipeline Editor

The Pipeline Editor is a new feature in GrassMarlin 3.3. Prior versions had only a simple, immutable Pipeline. This was a significant change that was the driving force behind the decision to start development on 3.3 from scratch. Changing the Pipeline allows for direct interaction with external systems, allows for the integration of new sources of metadata, and can result in a dramatic change in the system resources required by the application (either more or less, depending on the changes).

The Pipeline Editor (which can be accessed from the "Tools" menu) allows a user to create a new Pipeline, as well as view existing Pipelines. The Pipeline Editor functions as a standalone window, in part because it modifies supplementary data files, it does not interact with the current Session.

The Pipeline Editor, being a brand-new feature, has a few quirks that need to be hammered out yet. How files are saved is a little odd and, while the underlying system is sound, the presentation leaves something to be desired, especially with regards to how naming of Pipelines is handled, as well as deriving a new Pipeline from an existing Pipeline.

Although the Pipeline Editor is described in detail later, the basic usage is as follows:

  • To create a new Pipeline, select the New menu item from the File menu and enter the name of the new Pipeline when prompted. Alternatively, loading a Pipeline or working on the default Pipeline is possible, but in these cases it must be Exported, rather than saved.
  • New entry Points and Stages can be added from a context (right-click) menu.
  • To remove existing links, hold Ctrl and click on the arrow at either end of the link. This will remove all connections to/from that arrow.
  • To add a new link, click one endpoint so that it is highlighted, then hold Shift and click on the other endpoint of the link.
  • It is not possible to modify the Pipeline associated with the current Session. Before content is Imported into a Session, the Pipeline can be selected. Once an Import begins, the Pipeline for that Session becomes fixed and cannot change. The Pipeline Editor does not modify the Session's Pipeline, it modifies the template that is used to construct that Pipeline when the first Import begins--once the Pipeline is created, changes to the Template have no effect.

7 Plugin Development

Plugins are Java files which can be loaded at runtime to add new behaviors to GrassMarlin, or to modify existing behaviors. Each Plugin must provide a class, named Plugin, within a package that matches the Plugin's name (e.g. the iadgov.ping Plugin provides the iadgov.ping.Plugin class). This class must implement the grassmarlin.plugins.IPlugin interface. The grassmarlin.plugins.IPlugin interface contains several additional interfaces, each of which addresses a specific behavior.

It is our goal to facilitate widespread Plugin development, but make no mistake--this is intended for experienced software developers. This is not a tutorial but reference documentation.
